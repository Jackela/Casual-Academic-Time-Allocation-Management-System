# 诚实改造指南

这个文档不是教你“怎么把别人的项目直接拿去交差”，而是教你如何把 CATAMS 当作参考案例，合法且诚实地做出一个属于你自己的课题版本。

## 1. 先看清边界

当前仓库的 [LICENSE](../../LICENSE) 仍然是 University of Sydney 专有许可。

所以在没有进一步确认授权之前，你不应默认自己有权：

- 直接复制源码
- 直接复制截图
- 直接复制文档
- 直接再分发仓库中的专有材料

更稳妥的做法是：学习系统结构、测试方法和文档组织方式，再自己重写和重实现。

## 2. 如果你真的要把它变成自己的课题，至少要改什么

必须改：

- 题目
- 项目名称
- 业务背景
- 用户角色命名
- 数据模型命名
- 界面截图
- README / 使用说明 / 架构说明
- 测试数据与演示数据

建议也改：

- 审批流细节
- 表单字段
- 状态命名
- 目录结构说明
- 页面布局与视觉风格
- 关键技术选型或部署方式

## 3. 诚实的改造方式是什么

更合理的路线是：

1. 先把这个项目当成“参考系统”，看清楚它如何组织角色、规则和测试。
2. 重新写你自己的需求背景和目标。
3. 用你自己的业务术语重新建模。
4. 自己实现关键页面和接口，而不是只改文字。
5. 用你自己的数据、截图、图表和测试报告替换原始证据。
6. 在文档里明确说明你参考了哪些公开资料或项目结构。

## 4. 哪些做法不应该做

不要这样做：

- 只改封面和仓库名，业务内容仍然完全一样。
- 直接保留原始截图，只换一个项目标题。
- 删除来源后宣称“完整原创”。
- 在老师或评审面前隐瞒你使用过的参考仓库。

## 5. 推荐的引用方式

如果你的课题确实受到了这个仓库启发，可以在文档中保留一句简短说明，例如：

> 本项目在系统结构、测试组织或交付文档方面参考了 CATAMS，但业务背景、需求、界面、数据和实现细节已按本课题重新设计。

## 6. 一个更安全的改造思路

如果你只是想借鉴思路，不想碰许可风险：

- 把它当作架构与测试参考，而不是代码模板。
- 重新写需求说明和页面流程。
- 重新建立自己的仓库与提交历史。
- 自己画图、自己截图、自己写演示材料。
